第 4 章 系统设计与实现

4.1 功能模块设计

本系统围绕校园失物招领流程进行功能划分，以用户端与管理员端双视角构建模块体系。用户端提供失物信息发布、招领信息发布、分类检索、物品详情查看、认领申请提交与进度查询，并在个人中心维护资料与历史记录；管理员端负责用户审核、物品分类维护、失物与招领信息管理、认领申请审核与状态流转，同时通过通知发布与消息推送保持沟通闭环。文件上传与图片展示作为通用支撑能力，贯穿物品信息采集与证据提交环节，各模块共享用户、分类与状态数据，形成发布—认领—审核—通知的闭环。如图 4-1 所示。

图 4-1 系统功能结构图

4.2 数据库设计

数据库设计以 lost_found_db.sql 为依据，结合系统功能与数据流转需求进行建模，重点保障物品信息与认领流程的可追溯性。数据库结构由用户、物品、认领与通知四类核心实体组成，并通过分类表统一管理物品类别，使信息发布、状态更新与审核记录能够在同一数据模型内闭环维护。

4.2.1 E-R 设计

在 E-R 设计中，user 作为主体实体，与 lost_item、found_item 形成一对多关联，体现用户发布记录；item_category 与两类物品表形成一对多关联，用于分类管理；claim_application 作为认领流程的桥接实体，分别关联 user 与物品记录，并通过 item_type 区分失物与招领场景；notification 与 user 建立一对多关联，用于承载审核结果与系统提醒。实体关系围绕发布、认领、审核与通知展开，如图 4-2 所示。

4.3 前端功能实现

4.3.1 失物信息页面及功能

失物信息页面以列表展示为核心，提供分类筛选、关键字检索与时间排序入口，列表中呈现物品标题、丢失地点、丢失时间与状态标签，帮助用户快速定位线索并进入详情查看。页面支持分页与快速跳转，保持信息浏览的连续性。

图 4-3 失物信息列表图

详情页面集中展示物品描述、图片与联系方式，便于拾获者核对线索并提交认领说明；发布用户可通过编辑入口更新信息与状态，确保失物信息与实际情况同步。

图 4-4 失物详情展示图

4.3.2 招领信息页面及功能

招领信息页面面向拾获者发布内容的集中展示，列表中突出拾取地点、拾取时间与物品分类，并提供条件筛选与检索功能，便于失主按场景快速检索招领记录。

图 4-5 招领信息列表图

详情页面展示招领物品的外观描述与图片凭证，并标注联系方式与当前认领状态，失主可在此提交认领申请或补充说明，形成线上申请与线下交接的衔接。

图 4-6 招领详情展示图

4.3.3 认领申请页面及功能

认领申请页面分为“我的申请”与“待我审核”两类视图，支持按状态筛选、查看申请详情与时间记录，并提供取消、通过或拒绝等操作入口，使申请流转与审核反馈保持可视化与可追溯。

通知消息页面集中展示系统通知、申请通知与审核结果，列表中通过已读状态与类型标签区分不同提醒，用户可快速定位与自身认领流程相关的信息，保证反馈闭环及时送达。

个人中心页面提供用户资料维护与账号安全配置，涵盖头像上传、姓名与联系方式更新、性别与邮箱维护，并支持密码修改与校验，确保用户身份信息与认领沟通渠道保持准确可靠。

4.4 后端功能实现

4.4.1 用户与权限管理功能

后台用户管理围绕账号审核、角色分配与状态维护展开，UserController 负责用户列表、详情查询与状态更新接口，配合 Spring Security 与 JWT 完成登录态校验与权限控制。管理员可在管理端查看用户基础资料与认证状态，对异常账号执行冻结、禁用或审核通过操作，确保发布与认领行为在可信身份下进行。

图 4-10 用户管理列表图

4.4.2 失物与招领信息管理功能

失物与招领数据由 LostItemController 与 FoundItemController 统一管理，后台支持按分类、时间与状态筛选信息，对违规或过期条目执行下架与状态更新。管理员在审核页面可查看物品描述与图片凭证，必要时补充审核备注，使信息发布流程与状态流转保持一致性与可追溯性。

图 4-11 失物与招领信息管理图

4.4.3 认领申请与通知管理功能

认领申请由 ClaimApplicationController 驱动，管理员根据申请说明与证据材料进行审核，审核结果写入申请记录并同步更新物品状态。NotificationController 负责生成系统消息与审核通知，用户在前端收到状态变更提醒，形成“申请—审核—通知”的闭环。
