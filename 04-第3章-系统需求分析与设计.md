第 3 章 系统需求分析与设计

3.1 可行性分析

3.1.1 技术可行性

系统采用 Vue 3、Vite 与 Element Plus 构建前端，配合组件化与路由机制实现页面拆分与角色导航；后端使用 Spring Boot 3.2、MyBatis-Plus、Spring Security、JWT 与 MySQL 形成稳定的服务体系，能够覆盖登录鉴权、信息发布、认领审核、通知推送与文件上传等核心流程。技术栈均为成熟的开源方案，社区生态完善，框架兼容性与部署路径清晰，便于在校园服务器或云主机上快速落地。

3.1.2 经济可行性

本系统以开源技术为核心，开发阶段无需额外授权费用，部署可使用现有校园机房或轻量云主机完成，软硬件投入可控。数据规模主要集中于失物、招领、认领与通知记录，存储与带宽压力有限，运维成本低，能够在保证服务质量的同时保持较高的投入产出比。

3.1.3 操作可行性

系统围绕“发布—检索—认领—审核—通知”形成清晰流程，前端提供分类筛选、时间地点检索、详情浏览与个人中心管理等入口，用户学习成本低。后台通过表格化管理与状态流转完成审核与统计，管理员可在短时间内掌握操作要点，满足日常管理与突发情况处理需求。

3.2 系统功能需求

3.2.1 用户主要功能

用户端需支持失物信息发布与招领信息发布，包含标题、分类、时间地点、描述与图片上传等要素，并可按分类、时间与关键字进行检索浏览。用户可对物品详情发起认领申请，跟踪申请状态与审核结果，同时在个人中心查看历史发布记录与收藏信息，接收系统通知与审核消息，形成完整的物品流转闭环。

3.2.2 管理员主要功能

管理员需具备用户信息管理、物品分类维护、失物与招领信息审核、认领申请处理与通知发布能力，可对异常或违规内容进行下架与状态调整。系统还应提供统计与查询功能，辅助管理人员了解高频遗失区域、物品类别与处理时效，为校园服务优化与安全治理提供数据支撑。

3.3 非功能性需求分析

3.3.1 性能需求

系统需支持校园高峰期的并发查询与信息发布，列表与检索接口在常规负载下保持秒级响应，分页与筛选应避免大范围全表扫描，保证多用户同时访问时页面加载稳定。

3.3.2 安全性需求

系统采用 JWT 与角色权限控制实现访问隔离，密码需加密存储，接口请求需校验登录态与权限范围；上传图片与附件需要类型与大小限制，避免恶意文件进入系统；敏感信息展示需最小化原则，保护用户隐私并满足校园安全管理要求。

3.3.3 易用性与可维护性需求

界面应保持统一布局与清晰操作指引，重要操作提供确认与提示，降低误操作风险；后端模块按业务分层组织，日志记录、异常处理与配置管理清晰，便于后续功能扩展与运维排查，保证系统长期可维护性。

3.4 系统用户用例分析

本系统围绕校园失物招领业务，将用例按角色划分为学生（失主）、拾获者与管理员三类，以保证发布、认领、审核和通知形成闭环。学生侧重点在于浏览信息、提交认领与跟踪结果；拾获者侧负责发布招领、补充物品线索并维护状态；管理员侧承担审核、分类维护与消息发布，保障信息准确性与处理效率。

学生用例图以失主为主体，涵盖注册登录、分类检索、查看失物与招领详情、提交认领申请、上传证明材料、查看申请进度与历史记录、个人信息维护等操作，并通过通知模块获取审核结果与状态变更，确保认领流程可追踪。
